Diffusion is not one forward pass

One image = many timesteps through the denoiser — closer to a loop than a single matmul festival.

The efficiency insult

You pay for dozens to hundreds of full U-Net evaluations per sample. LLM “prefill vs decode” intuition only gets you halfway — here the expensive bit is a sequential denoise loop, repeated until the sampler says stop.

Rough mental model
wall time ≈ T steps × ( U-Net(latent, t, cond) + overhead )

Quantization mostly squeezes the middle term — same T, cheaper per step.

What users feel

Seconds to tens of seconds waiting. Power draw. VRAM spikes. Queue depth when you batch traffic.

What you actually optimize

Per-step matmul and memory traffic — and whether your product can live with fewer steps or smaller latents.

Land the emotional truth: latency is steps × per-step cost. Quantization attacks the per-step term.

Latent diffusion, decomposed

Briefing vocabulary: forward noising (Gaussian noise until structure collapses) vs. backward denoising (U-Net predicts noise step-by-step). Stable Diffusion ties that loop to three modules:

Text encoder

CLIP-style: prompt → conditioning vectors. Important, but not always the FLOP king.

VAE (autoencoder)

Pixels ↔ latent tiles. Usually cheaper than the main denoiser at typical resolutions.

U-Net denoiser

The iterative workhorse. This is where Quark’s tutorial spends its quantization budget.

pixels —[encode]→ latents —[U-Net × T steps]→ latents —[decode]→ pixels
AMD’s doc walks SD components explicitly; use this slide to anchor “what we’re allowed to ignore first.”

Why weight quantization on the U-Net

Tutorial choice, engineering choice: quantize U-Net weights first — that’s where the tutorial expects the most performance upside vs. text encoder or VAE.
  • Smaller weights → less DRAM traffic → faster matmuls on friendly kernels.
  • You keep the same sampling schedule; you’re not redesigning the sampler (yet).
  • Activations still matter — but weight-only is a sharp first lever.
If someone asks “why not FP16 everywhere?” — answer: you can, but you’re leaving bandwidth on the table.

What AMD Quark is (no mythology)

Quark 0.11.1 (briefing baseline) is AMD’s model optimization toolkit: PyTorch and ONNX paths, multiple integer and float-low-bit dtypes, export — quantization as something you run in a lab notebook, not a slide-deck fairy tale.

You bring

A real model (here: diffusers pipeline), a GPU (ROCm or CUDA per docs), and calibration data.

Quark brings

Specs, layer configs, a ModelQuantizer, export hooks — the boring glue that ships.

Official write-up: AMD Quark open-source announcement · Docs hub: quark.docs.amd.com

Cite AMD’s own article for “what/why open” — keeps you out of fanboy territory.

Quantization in one breath

Idea: store weights (and sometimes activations) in fewer bits with scales/zero-points — trading representational headroom for speed and footprint.

What breaks

Outlier-heavy layers, sharp attention stats, anything that needed FP dynamic range “for free.”

What saves you

Calibration (representative inputs), symmetric vs asymmetric choices, and sometimes FP8 instead of INT8.

Point to Quark’s intro doc for bit-level intuition if the room wants homework.

The Quark API shape you’ll actually type

Pattern from the diffusion tutorial — slightly cleaned for slides:

Int8PerTensorSpec(observer_method="min_max", scale_type="float", is_dynamic=False, symmetric=True, round_method="half_even") → QLayerConfig(weight=…) → QConfig(global_quant_config=…) → ModelQuantizer(config)

Spec

Answers “how are we snapping weights to low-bit grids?”

Quantizer

Answers “run that policy across modules and give me a model I can drop back into the pipeline.”

Don’t read the spec string aloud — it’s wallpaper. Emphasize observer + symmetric INT8 defaults.

Precision ladder (briefing chart, lived experience)

Strategic U-Net weight quantization — quality vs. payoff, per the briefing’s synthesis:

FP16

Quality: baseline (high). Perf: standard reference.

FP8

Quality: near-identical to FP16 in practice. Perf: high upside.

INT8

Quality: slight variations — smaller representable range. Perf: very high.

Briefing verdict: FP8 is the sweet spot for “fast + hard to see the damage”; INT8 is the speed demon that demands adult supervision on calibration.

Code fork: Int8PerTensorSpec vs FP8E4M3PerTensorSpec — same Quark skeleton, different pain/return curve.

Tutorial side-by-side grids back the FP8 story; use them if anyone challenges “near-identical.”

Baseline: honest FP16 (or BF16) generation

Tutorial starts from a real diffusers pipeline — you need a reproducible anchor.

pipe = StableDiffusionPipeline.from_pretrained(model_id, torch_dtype=torch.float16)
pipe = pipe.to(device)
images = pipe(prompt=..., num_inference_steps=50, guidance_scale=7.5, generator=seeded).images
Why bother: if you can’t A/B at the same seed + steps + guidance, you’re arguing about weather, not quantization.
Mention fixed generator seed from the tutorial — it’s the adult-in-the-room move.

Calibration & recalibration (briefing workflow)

Quantization eats accuracy. The briefing frames recovery as rigorous recalibration — representative data, then weight adjustment, then proof against a baseline. Skip any leg of that tripod and you’re doing vibe-based ML.
  1. Calibration data — e.g. LAION-400M class datasets; HF login / gated access per docs.
  2. Weight adjustment — observers + Quark’s quant path recover what precision lopped off (briefing calls this “finetuning” in the broad sense; the tutorial implements it as calibrated quantize_model).
  3. Image A/B — same prompt distribution, seeded runs, honest visual diff vs. FP16.

Production move: swap LAION for your domain tensors when you can — the briefing still applies: representative, diverse, not one lazy folder of thumbnails.

laion/laion400m: access friction + policy — say it out loud. Clarify in Q&A: PTQ calibration ≠ full retrain unless you add that.

Swap in the quantized U-Net

model = pipe.unet
quantizer = ModelQuantizer(quant_config)
quant_model = quantizer.quantize_model(model, calib_dataloader)
pipe.unet = quant_model

Don’t confuse outputs

If you quantize the wrong submodule, you’ll “succeed” and still measure nothing useful.

Do measure

Wall time per image, VRAM high-water, and a blind A/B folder your team actually argues over.

Freeze + ONNX export

Tutorial path: freeze the quantized module, dummy inputs, then export_onnx.

frozen = quantizer.freeze(quant_model)
export_onnx(frozen, export_path, input_args) → quark_model.onnx
Interop story: ONNX moves graphs between frameworks (e.g. PyTorch → ONNX Runtime). The briefing explicitly calls out deployment across AMD Ryzen AI and AMD Instinct-class accelerators — plus the usual ORT / edge runtime suspects.

HF Optimum ONNX guides help when the runner isn’t Torch anymore.

Dummy latents / timestep / encoder hidden states must match the architecture you export.

Netron will show Q/DQ nodes — read that ethically

Briefing + tutorial honesty bomb: Quark often uses pseudo-quantization: lower-precision values carried inside original wider dtypes (think trailing-zero padding), so file size may not shrink until you land in a format that truly packs smaller types. Netron shows explicit quantization / dequantization blocks in the graph.

Visualize with Netron (netron.start("export_path/quark_model.onnx") in the docs) — zoom until your eyes hurt; the graph is huge.

Operator checklist (from the docs school of hard knocks)

Environment

Briefing is blunt: for diffusion, a GPU is mandatory — ROCm (AMD) or CUDA (NVIDIA). Windows: plan on WSL/Ubuntu per AMD’s setup notes.

PyTorch builds

Match ROCm or CUDA wheels to your stack; Quark is not a substitute for a broken driver story.

HF access

Models (e.g. SD v1-5) + calibration sets (e.g. LAION-400M) need a Hugging Face login when gated. Point HUGGINGFACE_HUB_CACHE / dataset cache somewhere sane if you’re pulling big iron.

Model IDs

Changing checkpoint shapes means revisiting ONNX dummy tensors — obvious, often forgotten.

Pull WSL + ROCm/CUDA notes straight from AMD’s installation section — saves support debt.

When Quark (or quantization) is the wrong hero

  • You’re batch-size 1 and bound by VAE or I/O — fix the actual bottleneck first.
  • Your “quality bar” is luxury-brand photography and INT8 is audibly ugly — don’t gaslight the art director.
  • You need true on-disk tiny weights for edge — plan compression + format goals explicitly.
Sharp takeaway: Quark gives you levers; it doesn’t revoke physics.

Summary

Briefing closes with three claims — and they’re still true on a whiteboard: FP8 balances speed with barely visible loss; INT8 is fastest but wants careful calibration; targeted U-Net optimization is the efficient compression path in latent diffusion.

Why

Iterative denoise multiplies cost; low-bit U-Net weights cut the per-step tax.

How

Spec → QConfig → ModelQuantizer → recalibrated U-Net in the pipeline.

Proof

Images, wall time, ONNX + Netron — receipts in that order.
